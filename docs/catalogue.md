# Emily UI Catalogue

**Last Updated:** 2026-07-05

Internal catalogue of the Emily UI system: what exists, where it lives, and how to use it. Plain markdown — this is not a public docs site.

## The ladder

```text
tokens / ions → manifest → atoms → molecules → organisms → templates → pages/starters → catalogue
```

| Layer | Lives in | What it is |
|---|---|---|
| Tokens/ions | `emily-css` (generated from `emily.config.json`) | Colour scales, semantic role tokens, spacing, type, radius, shadows, focus ring |
| Atoms | `emily-components/atoms/` | Smallest usable pieces: buttons, inputs, typography, media, feedback, layout primitives |
| Molecules | `emily-components/molecules/` | Atoms composed into units: form fields, cards, alerts, breadcrumb, pagination |
| Organisms | `emily-components/organisms/` | Molecules composed into page sections: heroes, footers, pricing, testimonials |
| Templates | `emily-components/templates/` | Section orders for whole pages, composed from organisms |
| Pages/starters | `emily-starters/` | Working sites: template compositions + their own `emily.config.json` + built CSS |

Legacy note: `patterns/` predates the atom layer (Phase 1 composition examples) and stays until molecules/organisms formally replace it.

## How to use each layer

1. Run `emily-css init` + `emily-css build` in your project — everything below consumes the generated classes.
2. **Starting a site?** Copy the closest starter from `emily-starters/` (see `starter-selection-guide.md`), re-brand via its `emily.config.json`, rebuild.
3. **Adding a section?** Copy from `organisms/` (or a whole page order from `templates/`), replace placeholder copy.
4. **Building something custom?** Compose from `molecules/` and `atoms/`. Every class in every example is generated by `emily-css build` — no other CSS exists.
5. New CSS never goes in example files — it goes in `emily-css/src/generators/patterns.js`.

## Status rules (read before using anything)

Full definitions in `manifest-guide.md`; current tables in `build-status.md`.

- **Use `built` items now.** They have real examples and real generated CSS.
- **Do not use `needs-js` items** until the matching emily-js behaviour exists. There is no JS layer wired yet.
- **Do not treat `needs-a11y-review` items as production-ready.** They await manual AT testing (see `accessibility-status.md`).
- **Do not use `deferred` items as promises.** They are parked, not scheduled.
- `needs-css` / `planned` items have no usable example yet — nothing to copy.

## Agent guidance

For any AI agent (or script) working on or with Emily UI:

1. **Read `component-manifest.json` first.** It is the single source of truth for what exists.
2. **Never guess component availability.** If it isn't `status: "built"` in the manifest, it doesn't exist for composition purposes.
3. **Only compose from built dependencies.** Follow each entry's `dependencies` array; if a dependency isn't built, the composition isn't buildable.
4. **Take markup from `examplesPath`,** never from memory.
5. **Brandbadger is a consumer, not the source of truth.** Nothing Brandbadger-specific (copy, colours, scheme names, business assumptions) belongs in Emily UI.
6. Statuses are honest. Do not mark anything `built` without a real example file and real generated CSS.

## Images

Examples use local placeholder paths only (`/images/placeholders/…` in components, relative `images/placeholders/…` in starters). No third-party image hosts, ever, in reusable examples. Full rules: `../README.md` → Image placeholders.
