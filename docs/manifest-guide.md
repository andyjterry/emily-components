# Component Manifest Guide

**Last Updated:** 2026-07-05

`component-manifest.json` is the exhaustive map of the Emily UI system — every layer, every planned component, honest statuses. It is the file humans, agents, and (per the branching init spec) future CLI tooling read before doing anything.

## Top-level shape

```json
{
  "name": "Emily UI component manifest",
  "lastUpdated": "2026-07-05",
  "layers": ["token", "atom", "molecule", "organism", "template", "page"],
  "statuses": ["planned", "built", "needs-css", "needs-js", "needs-a11y-review", "deferred"],
  "counts": { "<layer>": { "total": 0, "<status>": 0 } },
  "components": [ ...entries ]
}
```

`counts` is derived data. Never hand-edit it — recount after any status change (the update scripts do this automatically; `docs/generate-build-status.js` reads it).

## Entry shape

```json
{
  "id": "molecule-form-molecules-radio-group",
  "name": "Radio group",
  "layer": "molecule",
  "category": "form-molecules",
  "status": "built",
  "dependencies": ["atom-inputs-radio"],
  "variants": [],
  "states": [],
  "accessibilityNotes": "Fieldset + legend names the question; single name attribute per group.",
  "filePath": "emily-css/src/generators/patterns.js",
  "examplesPath": "emily-components/molecules/forms.html"
}
```

| Field | Meaning |
|---|---|
| `id` | Stable: `<layer>-<category>-<slug(name)>`. Referenced by `dependencies` — never rename casually |
| `status` | See below. The only field that gates usage |
| `dependencies` | ids this item is composed from. Must all exist; must all be `built` before this item can honestly be `built` |
| `accessibilityNotes` | The a11y contract and/or the reason for a blocked status |
| `filePath` | Where the implementation (usually CSS generator) lives |
| `examplesPath` | Where the copy-paste markup lives. `built` without an examplesPath is invalid |

## Status meanings

| Status | Meaning | May you use it? |
|---|---|---|
| `built` | Real example file + real generated CSS exist | Yes |
| `planned` | Mapped, nothing usable yet | No |
| `needs-css` | Blocked on CSS not yet in an emily-css generator (reason in notes) | No |
| `needs-js` | Cannot work without JavaScript; emily-js behaviour not yet wired | No |
| `needs-a11y-review` | Static markup may exist but manual AT testing is outstanding | Not for production |
| `deferred` | Parked deliberately (third-party dependency, product decision pending) | No — and not a promise |

## Honesty rules

- `built` requires: example file exists, every class in it is generated by `emily-css build`, no JS required, no pending manual sign-off (FAQ items are the one flagged exception, with the caveat recorded in notes).
- A status change is a claim. Downgrade freely; upgrade only with evidence (example + validated classes, or completed manual test results).
- Layer discipline: molecules from atoms, organisms from molecules, templates from organisms, starters from templates. Don't reclassify to dodge a blocker.

## How tools consume it

- **Agents:** filter `components` by `status === "built"`, take markup from `examplesPath`, follow `dependencies` for auto-includes. Never offer anything else.
- **Docs:** `docs/generate-build-status.js` renders the status tables.
- **Future branching init** (`branching-init-spec.md`): question options = manifest queries (`layer` + `category` + `status`); caveats surfaced from `accessibilityNotes`.

## Updating safely

1. Edit via script where possible (status changes in batches, counts recounted, dependency ids validated).
2. Validate: `node -e "JSON.parse(require('fs').readFileSync('component-manifest.json','utf8'))"`.
3. Regenerate `docs/build-status.md`.
4. Commit manifest + regenerated docs together.
